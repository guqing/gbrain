# Design: Post-Capture Insight Hook — 存入即启发

Generated by /office-hours on 2026-04-12
Branch: feat/v0.6-unified-extraction
Repo: guqing/gbrain
Status: APPROVED
Mode: Builder

## Problem Statement

知识存进 gbrain 之后处于"沉睡"状态。用户不知道该搜索什么，随着时间推移忘记已有内容，无法扫描全库。

更深层的问题：**不在于找不到已知的答案，而在于不知道该问什么问题。** 最有价值的认知跃升来自"我没想到 X 和 Y 有关"这类非显而易见的连接。

在 agent 时代，执行已经被 AI 代替，稀缺的是**好想法**。每一次存入新知识都应该是一次"放大"机会：系统用已有知识库为这条新知识生成启发，推动用户继续学习和实践。

学习路径：实践 → 启发 → 学习 → 启发 → 实践。gbrain 应该成为这个循环里的主动参与者。

## What Makes This Cool

存进去一条，弹出来 3 条你没想到的问题或连接。不是"相关文档列表"，是"你刚存的这件事，让我想到..."。

冷热对比：

**现在（冷）：**
```
exo put "React Suspense 让组件在数据准备好之前挂起"
→ 存进去，没有任何反馈
```

**目标（热）：**
```
exo put "React Suspense 让组件在数据准备好之前挂起"
→ 存储成功

💡 gbrain insight:
你 3 个月前写过：「人类思维是顺序的，异步是反直觉的」
问题：Suspense 是在解决认知模型问题，而不只是技术问题吗？

还有一条：你有一篇关于 Loading States UX 的笔记——
它们在用户心智模型上可能是同一个问题。
```

## Constraints

- 不破坏现有工作流（存就是存，启发是附加的）
- 启发必须是非显而易见的，否则是噪声，比没有更差
- 冷启动：候选相邻页少于 3 个时，跳过（行为驱动，不依赖总页数）
- 延迟可接受范围：2-4 秒（LLM 调用），不需要流式输出
- 不需要 UI，CLI 输出就够
- `capture` 是"无 LLM < 100ms"设计，**不触发 insight**；只在 `put` 触发
- 非 TTY 环境（管道/批量导入）时自动关闭，避免静默触发 N 次 LLM 调用

## Premises

1. 问题不在于"存了找不到"，而在于"不知道该问什么"——最有价值的发现来自未被预期的连接
2. gbrain 已有向量嵌入 + FTS5 + graph + compile LLM pipeline，缺的是触发机制和 prompt
3. 最小可验证路径：put/capture 后自动触发，返回 3 条连接或问题，零额外操作
4. 启发质量是成败关键：低质量输出比没有更差（用户强调了这一点）

## Approaches Considered

### Approach A: 存入即触发（Post-capture Insight Hook）— 已选

在 `exo put` / `gbrain capture` 写入后，自动：
1. 用新内容向量找前 5 个语义相邻的页
2. 构造 prompt：给 LLM 新知识 + 相邻页摘要，要求生成非显而易见的连接或问题
3. CLI 输出，用户看到即可（不需要确认）

Effort: S（几天）
Risk: 低
复用: sqlite-engine.ts 向量搜索, dispatch.ts LLM 调用

### Approach B: gbrain spark 命令（主动触发）

独立命令，用户主动调用，可传入主题，生成"思维链条"。
Effort: M（1-2周），Risk: 中

### Approach C: 图谱弱连接发现（Graph-aware Synthesis）

找语义距离"中等"的节点（不是最近邻，是有趣距离），生成"桥接假说"。
Effort: L（3-4周），Risk: 高

## Recommended Approach

**Approach A：存入即触发。**

理由：
1. 验证路径最短——如果几次操作内没有"哦！"的感受，先迭代 prompt，别建图谱
2. 复用现有 `sqlite-engine.ts` 向量搜索 + `dispatch.ts` LLM 调用，新代码量极少
3. 不破坏任何现有命令，是纯附加行为

**质量门控策略（关键）:**
- 不返回相关页列表，让 LLM 生成**问题**，而不是陈述
- Prompt 约束："只输出用户在存入这条知识时，不会自然想到的连接；找不到有价值的连接就输出空"
- 如果输出空，CLI 什么都不显示（不降低体验）

## Implementation Sketch

```typescript
// src/core/insight.ts
import type { SqliteEngine } from './sqlite-engine.ts';
import type { ExoConfig } from './config.ts';

// NOTE: accept optional fetchFn for testing (same pattern as compile/llm.ts callLlm)
export async function generateInsight(
  newPage: { slug: string; title: string; compiled_truth: string },
  engine: SqliteEngine,
  cfg: ExoConfig,
  fetchFn: typeof fetch = fetch,  // injectable for tests
): Promise<string> {
  const queryText = `${newPage.title}\n${newPage.compiled_truth.slice(0, 500)}`;

  // Use engine.hybridSearch directly (FTS5-only, null embedding) — no embed.api_key dependency.
  // hybrid.ts's hybridSearch() calls embed() which throws if embed.api_key is unset.
  // For insight, FTS5 keyword overlap is sufficient to find topically adjacent pages.
  const similar = engine.hybridSearch(queryText, null, {
    limit: 5,
    exclude_slugs: [newPage.slug]   // SearchOpts 字段名
  });
  if (similar.length < 3) return ''; // 冷启动保护

  const context = similar.map(r =>
    `[${r.slug}] ${r.title}: ${r.chunk_text.slice(0, 200)}`  // chunk_text, 非 snippet
  ).join('\n');

  const prompt = `You are a knowledge connector. The user just stored this note:

TITLE: ${newPage.title}
CONTENT: ${newPage.compiled_truth.slice(0, 800)}

Related notes already in their knowledge base:
${context}

Find 1-3 NON-OBVIOUS connections between the new note and existing notes.
Output QUESTIONS the user hasn't thought to ask, not summaries.
If no genuinely non-obvious connections exist, return empty string.
Same language as the new note. Max 120 words.`;

  // 复用 compile 配置做直接 LLM 调用（raw fetch + injectable fetchFn，同 compile/llm.ts 模式）
  const compileConfig = cfg.compile;
  if (!compileConfig?.api_key) return '';

  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), 3000); // 3s timeout
  try {
    const res = await fetchFn(`${compileConfig.base_url ?? 'https://api.openai.com/v1'}/chat/completions`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${compileConfig.api_key}`
      },
      body: JSON.stringify({
        model: compileConfig.model ?? 'gpt-4.1-mini',
        messages: [{ role: 'user', content: prompt }],
        max_tokens: 300,
        temperature: 0.7
      }),
      signal: controller.signal
    });
    const data = await res.json() as { choices: Array<{ message: { content: string } }> };
    return data.choices?.[0]?.message?.content?.trim() ?? '';
  } catch {
    return ''; // timeout 或网络错误静默跳过
  } finally {
    clearTimeout(timer);
  }
}
```

# Part 2: Deep Research Layer

Generated: 2026-04-12 (extended session)
Status: APPROVED

## Problem Statement (Layer 2)

Layer 1 (Post-Capture Insight Hook) 解决了"启发生成"的问题，但启发只是一个问题或连接——它缺少后续。

用户看到启发之后的状态：**有好奇心，但没有行动路径。** 要研究这个问题，需要自己打开浏览器、搜索、阅读多篇文章、回到 gbrain 比对已有知识，耗时 30-60 分钟。

核心 gap：**启发产生的那一刻，研究的冲动最强。** 但研究路径太长，冲动消退了，启发又变成了死知识。

`exo research "<question>"` 把这条路径压缩到 60-120 秒，输出一篇可以直接阅读的研究报告，保存进 inbox 成为新的知识。

## What Makes This Cool (Layer 2)

启发 → 研究 → 知识入库，形成闭环。研究的结果本身又成为知识，下次 put 时可能再生成新启发。这是一个**知识复利**机制。

```
exo put "React Suspense 让组件在数据准备好之前挂起"
→ 存储成功

💡 gbrain insight:
你 3 个月前写过：「人类思维是顺序的，异步是反直觉的」
问题：Suspense 是在解决认知模型问题，而不只是技术问题吗？

  → Run: exo research "Suspense 是在解决认知模型问题吗"

# 用户觉得这个问题值得深究，运行：
exo research "Suspense 是在解决认知模型问题吗"

🔬 Researching: Suspense 是在解决认知模型问题吗 (depth=1)
  → Generating sub-queries...
  → Searching Level 1 (4 sub-queries)...
  → Synthesizing research article...

# --- 输出 500-1500 字的 Markdown 研究报告 ---
✓ Saved to inbox: research-1744444800-suspense-cognitive
  Run 'exo compile' to integrate into knowledge base.

# depth=2 完整研究模式：
exo research "Suspense 是在解决认知模型问题吗" --depth 2

🔬 Researching: Suspense 是在解决认知模型问题吗 (depth=2)
  → Generating sub-queries...
  → Searching Level 1 (4 sub-queries)...
  → Searching Level 2 (3 follow-up questions)...
  → Synthesizing research article...
```

## Constraints (Layer 2)

- 不改变 Layer 1 的 insight 生成流程，research 是独立命令
- 外部依赖：Exa API（免费 1000次/月，语义搜索）+ 现有 cfg.compile.* LLM config
- 运行时间 60-120 秒可接受，但需要进度提示（不能无反应等待）
- 输出语言跟随输入语言（中文问题 → 中文报告）
- 报告自动保存进 inbox（除非 --no-save）
- 失败时 graceful：web search 失败降级到纯 gbrain 知识 + LLM 合成

## Premises (Layer 2)

1. 深度研究是耗时操作（60-120s），应该是用户显式触发，不是 put 时自动运行的
2. 研究结果保存进 gbrain inbox，让 compile 处理后成为知识库的一部分（研究本身变知识）
3. web search 需要外部 API（Exa），用户配置一个 key，这是可接受的额外依赖
4. 输出格式是结构化 Markdown 长文（500-1500 字），有引用来源，类似研究报告

## Approaches Considered (Layer 2)

### Approach A: 单次合成（Minimal）

单次 web search + 单次 gbrain 检索，一次 LLM 合成调用。
Effort: S | Risk: Low | 质量：中等（覆盖面有限）

### Approach B: 多阶段递归研究（Ideal）— 已选

LLM 分解问题 → 多个子查询并行搜索（web + gbrain）→ **相关性评估 → learning 提取 + follow-up questions** → 可选递归到 depth=2 → LLM 综合合成
Effort: M | Risk: Med | 质量：高（递归深度可控，depth=1 快速 / depth=2 完整）

### Approach C: 研究队列（spark 编号 + 持久化）

需要修改 spark、持久化历史到 db。Effort: L | Risk: High | 留 v0.8

## Recommended Approach (Layer 2): B + Exa API（递归研究）

**理由：** A 和 B 工程量差距主要是"把 1 次搜索改成并行 + 递归"，边际成本低，但输出质量差距大。

相关性评估 + learning 提取（来自 trigger.dev deep research 实践）是质量提升的核心：不把原始搜索结果直接丢给 LLM，而是先过滤无关内容、再提炼洞见，最终合成质量显著更高。

depth=1（快速，约 30s）适合日常使用；depth=2（完整递归，约 90s）用于真正需要深挖的研究。

**Web Search 方案对比：**

| 方案 | 免费额度 | 搜索类型 | 输出质量 |
|------|---------|---------|---------|
| **Exa API** | 1000次/月 | **语义搜索 + live crawling** | 高（研究型查询最优）|
| Tavily API | 1000次/月 | 关键词+摘要 | 高（信息检索场景）|
| Brave Search | 2000次/月 | 关键词 | 中 |
| Perplexity Sonar | 有限 | LLM 合成 | 高但贵 |

推荐 **Exa API**：语义搜索对"研究型"问题（概念理解、模式发现）比关键词搜索更准确，正是 gbrain 的核心场景。配置在 `~/.exo/config.toml` 的 `[research]` 段落。

### 3. 命令入口（src/commands/research.ts）

```typescript
import { defineCommand } from 'citty';
import { openDb, resolveDbPath } from '../core/db.ts';
import { SqliteEngine } from '../core/sqlite-engine.ts';
import { loadConfig } from '../core/config.ts';
import { generateInboxSlug } from '../core/utils.ts';
import { deepResearch } from '../core/research.ts';

export default defineCommand({
  meta: { name: 'research', description: 'Deep-research a topic using exo knowledge base + web search' },
  args: {
    db:       { type: 'string',  description: 'Path to brain.db' },
    question: { type: 'positional', required: true, description: 'Research question or insight to investigate' },
    'no-save': { type: 'boolean', description: 'Print only, do not save to inbox', default: false },
    depth:    { type: 'string',  description: '1 (fast, ~30s) | 2 (deep recursive, ~90s)', default: '1' },
  },
  async run({ args }) {
    const cfg = loadConfig(args.db ? { db: args.db } : undefined);

    // Graceful degradation: if no Exa key, run in local-only mode (hybridSearch only, no web results)
    const hasExa = Boolean(cfg.research?.exa_api_key);
    if (!hasExa) {
      console.log('ℹ No Exa API key configured — running in local-only mode (knowledge base only).');
      console.log('  Add to ~/.exo/config.toml to enable web search:\n  [research]\n  exa_api_key = "your-key"\n');
    }

    const depth = args.depth === '2' ? 2 : 1;
    const engine = new SqliteEngine(openDb(resolveDbPath(args.db)));
    console.log(`🔬 Researching: ${args.question} (depth=${depth}${hasExa ? '' : ', local-only'})`);

    const article = await deepResearch(args.question, engine, cfg, depth);
    console.log('\n' + article.content);

    if (!args['no-save']) {
      const slug = generateInboxSlug(article.content || args.question);
      try {
        engine.putPage(slug, {
          // NOTE: type:'research' requires adding 'research' to PageInput.type union in types.ts
          // This prevents exo compile from treating research reports as regular inbox notes.
          // Until types.ts is updated, use type:'inbox' but track via slug or metadata.
          type: 'inbox',
          title: article.title.length > 80 ? article.title.slice(0, 77) + '...' : article.title,
          compiled_truth: article.content,
        });
        console.log(`\n✓ Saved to inbox: ${slug}`);
        console.log(`  Run 'exo compile' to integrate into knowledge base.`);
      } catch (e) {
        const msg = e instanceof Error ? e.message : String(e);
        console.error(`  [error] Failed to save research report: ${msg}`);
      }
    }
  }
});
```

### 4. 核心研究逻辑（src/core/research.ts）

**Performance Notes（LLM 调用预算）：**

| 步骤 | 调用数 | 并发 | 预估耗时 |
|------|--------|------|---------|
| Sub-query 分解 | 1 | — | ~2s |
| Level 1 evaluate+extract | N（3-4） | 并行 | ~8-10s |
| Level 2 follow-up（depth=2） | 最多3 | 并行 | ~6-8s |
| Synthesis | 1 | — | ~15-20s |
| **depth=1 合计** | 5-6 | — | **~25-35s** |
| **depth=2 合计** | 8-9 | — | **~45-70s** |

每次 LLM 调用有 30s 超时保护。depth=1 是日常用途，depth=2 用于需要深挖的研究。

**Pipeline 设计（来自 trigger.dev deep research 实践）：**
```
问题
├── [Level 1] 生成子查询
│   每个子查询: Exa 搜索 + exo 本地检索
│             → 相关性评估（批量，1次LLM调用）
│             → 提取 learnings + follow-up questions
│
└── [Level 2, 仅 depth=2] follow-up questions 递归（并行，cap=3）
    每个: Exa 搜索 → 相关性评估 → 提取 learnings
    （不再继续递归，防止爆炸）


最终: 汇总所有 learnings → LLM 合成报告
```

```typescript
import type { SqliteEngine } from './sqlite-engine.ts';
import type { ExoConfig } from './config.ts'; // ExoConfig lives in core/config.ts, not types.ts

export interface ResearchArticle {
  title: string;
  content: string; // full Markdown
  sources: string[];
}

interface WebResult { title: string; url: string; content: string }
interface Learning { insight: string; source: string }

async function exaSearch(
  query: string,
  apiKey: string,
  maxResults = 5,
  fetchFn: typeof fetch = fetch,  // injectable for tests
): Promise<WebResult[]> {
  const res = await fetchFn('https://api.exa.ai/search', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'x-api-key': apiKey },
    body: JSON.stringify({ query, numResults: maxResults, useAutoprompt: true })
  });
  if (!res.ok) {
    console.error(`  [warn] Exa search failed: ${res.status} ${res.statusText}`);
    return [];
  }
  const data = await res.json() as { results: Array<{ title: string; url: string; text: string }> };
  return (data.results ?? []).map(r => ({ title: r.title, url: r.url, content: r.text ?? '' }));
}

// One LLM call per sub-query: evaluate relevance + extract learnings
// includeFollowUp: only request follow-up questions at depth=2 (avoids wasted tokens at depth=1)
async function evaluateAndExtract(
  query: string,
  webResults: WebResult[],
  grainResults: Array<{ slug: string; title: string; chunk_text: string }>,
  llmCall: (prompt: string, maxTokens: number) => Promise<string>,
  includeFollowUp: boolean = false,
): Promise<{ learnings: Learning[]; followUpQuestions: string[]; relevantUrls: string[] }> {
  if (webResults.length === 0 && grainResults.length === 0) {
    return { learnings: [], followUpQuestions: [], relevantUrls: [] };
  }

  const webSnippets = webResults
    .map((r, i) => `[W${i}] ${r.title}: ${r.content.slice(0, 350)} (${r.url})`)
    .join('\n');
  const grainSnippets = grainResults
    .map(r => `[G:${r.slug}] ${r.title}: ${r.chunk_text.slice(0, 250)}`)
    .join('\n');

  const evalPrompt = `Research query: "${query}"

Sources:
${webSnippets}
${grainSnippets}

Task:
1. Filter to only sources relevant to the query (relevance >= 6/10)
2. Extract 2-4 specific insights from relevant sources${includeFollowUp ? '\n3. Generate 1-2 follow-up questions worth researching deeper' : ''}

Output ONLY valid JSON:
{
  "relevantWebIndexes": [0, 2],
  "learnings": ["specific insight 1", "specific insight 2"]${includeFollowUp ? ',\n  "followUpQuestions": ["question worth digging into"]' : ''}
}`;

  try {
    const raw = await llmCall(evalPrompt, 400);
    const parsed = JSON.parse(raw);
    const idxs: number[] = Array.isArray(parsed.relevantWebIndexes) ? parsed.relevantWebIndexes : [];
    const learnings: Learning[] = (Array.isArray(parsed.learnings) ? parsed.learnings : [])
      .filter((x: unknown) => typeof x === 'string')
      .map((insight: string) => ({ insight, source: query }));
    const followUpQuestions: string[] = (Array.isArray(parsed.followUpQuestions) ? parsed.followUpQuestions : [])
      .filter((x: unknown) => typeof x === 'string');
    const relevantUrls = idxs
      .filter(i => i >= 0 && i < webResults.length)
      .map(i => webResults[i].url);
    return { learnings, followUpQuestions, relevantUrls };
  } catch {
    // JSON parse failed — return all as-is without filtering
    return {
      learnings: webResults.slice(0, 2).map(r => ({ insight: r.content.slice(0, 300) || r.title, source: r.url })),
      followUpQuestions: [],
      relevantUrls: webResults.map(r => r.url),
    };
  }
}

export async function deepResearch(
  question: string,
  engine: SqliteEngine,
  cfg: ExoConfig,
  depth: 1 | 2 = 1,
  fetchFn: typeof fetch = fetch,  // injectable for tests
): Promise<ResearchArticle> {
  const llmCfg = cfg.compile;
  const exaKey = cfg.research?.exa_api_key; // may be undefined — local-only mode if missing
  if (!llmCfg?.api_key) throw new Error('Missing compile.api_key — run: exo config set compile.api_key <key>');

  const maxResults = cfg.research?.max_results_per_query ?? 5;
  const llmBase = llmCfg.base_url ?? 'https://api.openai.com/v1';
  const llmModel = llmCfg.model ?? 'gpt-4.1-mini';
  const llmHeaders = {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${llmCfg.api_key}`
  };

  async function llmCall(prompt: string, maxTokens: number): Promise<string> {
    const controller = new AbortController();
    const timer = setTimeout(() => controller.abort(), 30_000);
    try {
      const res = await fetchFn(`${llmBase}/chat/completions`, {
        method: 'POST',
        headers: llmHeaders,
        body: JSON.stringify({
          model: llmModel,
          messages: [{ role: 'user', content: prompt }],
          max_tokens: maxTokens,
          temperature: 0.4
        }),
        signal: controller.signal
      });
      if (!res.ok) throw new Error(`LLM API error: ${res.status} ${res.statusText}`);
      const data = await res.json() as { choices: Array<{ message: { content: string } }> };
      return data.choices?.[0]?.message?.content?.trim() ?? '';
    } finally {
      clearTimeout(timer);
    }
  }

  async function exaSearchLocal(query: string): Promise<WebResult[]> {
    if (!exaKey) return []; // local-only mode: skip web search
    return exaSearch(query, exaKey, maxResults, fetchFn).catch(() => [] as WebResult[]);
  }

  // Step 1: 问题分解 → 3-4 个子查询
  process.stdout.write('  → Generating sub-queries...\n');
  let subQueries: string[] = [question];
  try {
    const sqRaw = await llmCall(
      `Break this research question into 3-4 focused web search queries.\nQuestion: "${question}"\nOutput ONLY a JSON array of strings. Example: ["query 1", "query 2", "query 3"]`,
      300
    );
    const parsed = JSON.parse(sqRaw);
    if (Array.isArray(parsed) && parsed.every((x: unknown) => typeof x === 'string')) {
      subQueries = parsed;
    }
  } catch {
    process.stderr.write('  [warn] Sub-query decomposition failed — using original question only.\n');
  }

  // Step 2: Level 1 — 并行搜索 + 相关性评估 + learning 提取
  process.stdout.write(`  → Searching Level 1 (${subQueries.length} sub-queries)...\n`);
  const allLearnings: Learning[] = [];
  const allSources: string[] = [];
  const allFollowUps: string[] = [];

  const level1Results = await Promise.all(
    subQueries.map(async (q) => {
      const [webResults, grainResults] = await Promise.all([
        exaSearchLocal(q),
        Promise.resolve(engine.hybridSearch(q, null, { limit: 3 }))
      ]);
      return evaluateAndExtract(q, webResults, grainResults, llmCall, depth === 2);
    })
  );

  for (const r of level1Results) {
    allLearnings.push(...r.learnings);
    allSources.push(...r.relevantUrls);
    allFollowUps.push(...r.followUpQuestions);
  }

  // Step 3 (depth=2 only): Level 2 — follow-up questions 递归
  if (depth === 2 && allFollowUps.length > 0) {
    // cap at 3 follow-ups to prevent explosion
    const followUps = allFollowUps.slice(0, 3);
    process.stdout.write(`  → Searching Level 2 (${followUps.length} follow-up questions)...\n`);
    const level2Results = await Promise.all(
      followUps.map(async (q) => {
        const webResults = await exaSearchLocal(q);
        return evaluateAndExtract(q, webResults, [], llmCall, false);
      })
    );
    for (const r of level2Results) {
      allLearnings.push(...r.learnings);
      allSources.push(...r.relevantUrls);
    }
  }

  // Graceful degradation
  if (allLearnings.length === 0) {
    process.stdout.write('  [warn] No learnings extracted — synthesizing from question only.\n');
  }

  // Step 4: LLM 综合合成（输入是 learnings，不是原始搜索结果）
  process.stdout.write('  → Synthesizing research article...\n');
  const learningBlock = allLearnings
    .map((l, i) => `${i + 1}. ${l.insight}`)
    .join('\n');

  const synthesisPrompt = `You are a research synthesizer. Write a comprehensive research article on:
"${question}"

Based on these extracted research insights:
${learningBlock.slice(0, 6000)}

Write a structured Markdown article (500-1500 words) with:
- A clear title (# heading)
- An executive summary (3 sentences)
- 3-5 main sections with headers
- Concrete insights and analysis, not just summaries
- A ## Sources section at the end listing relevant URLs

Write in the same language as the question. Focus on non-obvious connections and implications.`;

  let content: string;
  try {
    content = await llmCall(synthesisPrompt, 2000);
  } catch (e) {
    // synthesis timeout or API error — return a minimal article rather than crashing
    const errMsg = e instanceof Error ? e.message : String(e);
    process.stderr.write(`  [warn] Synthesis failed: ${errMsg}\n`);
    content = `# ${question}\n\n*Synthesis failed. Raw learnings:*\n\n${learningBlock.slice(0, 2000)}`;
  }
  const uniqueSources = [...new Set(allSources)];

  return { title: question, content, sources: uniqueSources };
}
```
