## Product Framing

v0.5 不是 “只为 ChatGPT 做的图片导入功能”。它包含两层：

1. **通用文件系统（source-agnostic）**
   - 文件存储、去重、页面关联
   - 图片描述
   - 文件描述的 FTS5 / vector 搜索
   - `attach / detach / files / describe`
2. **来源适配器（source adapter）**
   - `import-chatgpt` 是第一个实现
   - 后续 `import-dir` / `import-claude` / 其他导入器都应复用同一套底层文件能力
3. **通用 importer framework**
   - source scan → page emit → file attach → provenance record → import run report
   - `import-chatgpt` 只是 framework 的第一个 adapter，不允许把 ChatGPT 特殊逻辑泄漏进通用层

### Scope Boundary

- **v0.5 语义理解范围：仅图片**
  - image attachments: 支持描述、搜索、导入时自动挂接
  - non-image files: 允许作为通用文件附件存在，但不做内容提取 / OCR / transcript
- **v0.6 再支持非图片理解**
  - PDF text extraction / OCR
  - audio / video transcript
  - richer multimodal parsing

## What We're Building

gbrain 目前只处理纯文本 markdown。用户有一个 ChatGPT 导出：490 个对话，50 个含图片，538 张图像，6 个 PDF。这些内容无法进入知识库。v0.5 的目标不是做一个 GPT 专用附件补丁，而是先交付一套 **通用文件底座**，并用 ChatGPT 导入作为第一个适配器把图片场景跑通。

1. `gbrain attach <page-slug> <file>` — 通用文件挂接能力（与来源无关）
2. `gbrain describe <file-slug>` — **仅对图片** 用 vision LLM 生成描述（可被 FTS5 + vector 搜索）
3. `gbrain import-chatgpt <export-dir>` — 首个来源适配器：导入 ChatGPT 会话，并自动挂接其中的图片
4. 搜索层升级：图片描述与页面正文一起进入检索路径
5. 导入运行态升级：用户可以看到 import run 状态、失败项、以及可重试单元

### CLI Information Architecture

v0.5 的关键用户面不是网页，而是 CLI / MCP 输出。本版本所有关键命令都使用同一套三层信息架构：

1. **状态摘要**
   - 第一屏只回答：成功了吗，影响了多少对象，是否有失败/跳过/未匹配
2. **下一步动作**
   - 如果用户现在最该做什么，直接给出一条命令，不让用户自己猜
3. **详细明细**
   - 失败项、引用上下文、匹配原因、重试对象、JSON 结构化字段

ASCII flow:

```text
command result
   |
   +--> Summary
   |      status / counts / severity
   |
   +--> Next action
   |      one recommended command or "no action needed"
   |
   \--> Details
          table / list / JSON payload / provenance / snippets
```

Per-command hierarchy:

```text
imports <run-id>
  1. run status + completed/failed/retryable counts
  2. resume/retry command
  3. failed items table

files [page]
  1. total files + described/undescribed counts
  2. describe next step when needed
  3. file rows with provenance summary

search <query> / query <question>
  1. best mixed results grouped by result_kind
  2. why this matched
  3. parent page / provenance / raw snippet

get <slug>
  1. page title + type + freshness
  2. attached file summary
  3. full compiled_truth / timeline / attachment details
```

Design rule:
- human-readable output and `--json` output must carry the same hierarchy, not two different products
- default terminal view optimizes for scanability first, completeness second
- JSON / MCP output must preserve `summary`, `next_action`, and `details` as separate layers

### Interaction State Table

| Feature | Loading | Empty | Error | Success | Partial |
|---|---|---|---|---|---|
| `attach` | `Attaching {file}...` only if --describe triggers vision API call | — (always operates on a file) | `✗ Page not found: {slug}` or `✗ Unsupported file type` or `✗ Vision API failed` (exit 1) | `✓ Attached {slug} to {page}` (exit 0) | `✓ Attached {slug} to {page} — ⚠ description failed. Run: gbrain describe {slug}` (exit 0 with warning) |
| `detach` | — (instant DB op) | — | `✗ File not found: {slug}` or `✗ Page not found: {slug}` (exit 1) | `✓ Detached {slug} from {page}` (exit 0) | — |
| `import-chatgpt` | Show source scan progress and current item counts | `No conversations found in export directory.` + suggest checking export structure | `Import failed before any pages were written.` + show first blocking error | `Imported N conversations, attached M images.` | `Imported N conversations, K images unmatched, E conversations skipped.` + suggest `gbrain imports <run-id>` |
| `imports` | `Loading import run...` only for expensive detail reads | `No import runs yet.` + suggest first import command | `Could not load run {id}.` + suggest `gbrain imports --json` for inspection | Show run status and retryable counts | `Run completed with errors.` + show retry command before failed rows |
| `files` | `Loading files...` only when querying all files | `No attached files yet.` + suggest `gbrain attach` or `gbrain import-chatgpt` | `Could not list files.` + surface DB error summary | Show total files, described/undescribed counts | `Files attached, but X are still undescribed.` + suggest `gbrain describe --all` |
| `describe` | `Describing {slug}...` for single file; `Describing X/Y files...` for --all | `No image files need description.` + no-action-needed | `Description failed for X files.` + separate retryable vs permanent skips | `Described X files.` | `Described X files, Y failed, Z skipped by size/type.` + show retry command |
| `search` / `query` | `Searching...` only when work is visibly long-running | `No matching pages or described files found.` + suggest broader query or `gbrain describe --all` | `Search completed with degraded results.` + state which source failed | Show grouped best matches with why-matched snippets | `Page matches available, file search unavailable.` or inverse, never silent |
| `get` | none for normal local reads | `Page not found: {slug}.` + suggest `gbrain list` or `gbrain search` | `Could not render page.` + preserve raw failure reason | Show page content plus attachment summary | `Page loaded, but some attachments are missing metadata.` + show file slugs needing repair |

State design rules:
- empty states must always include one concrete next command
- partial states are first-class, never hidden under generic success copy
- degraded search must be visible in both human output and MCP metadata
- permanent skip vs retryable failure must use different wording

### User Journey Storyboard

| Step | User does | User feels | Plan must support |
|---|---|---|---|
| 1 | Runs `import-chatgpt <export-dir>` | hopeful but uncertain | immediate source summary so user knows the export was recognized correctly |
| 2 | Watches scan/import progress | patient if progress is real, suspicious if static | visible progress, item counts, and whether failures are accumulating |
| 3 | Sees some unmatched files / skipped conversations | worried about silent data loss | partial-success summary with retryable vs permanent categories |
| 4 | Import finishes | asks “so what changed?” | status summary plus one recommended next action |
| 5 | Verifies the archive is now useful | wants proof, not theory | example `search` / `query` command or direct follow-up action |
| 6 | Runs `describe --all` later | cautious about cost/time | preflight with candidate counts, skip reasons, and expected work size |
| 7 | Searches for an imported image/topic | expects obvious why-matched context | grouped result hierarchy, provenance, and snippet source |

Design rule:
- every long-running command must end with a **value verification tail**
- the tail always answers: what changed, what still needs work, how to verify usefulness now
- recommended proof steps must be concrete commands, not generic advice

### DESIGN.md Alignment

v0.5 新增命令和 retrieval surfaces 必须继承 `DESIGN.md` 已定义的 CLI 语言：

- 默认输出为 human-readable plain text / markdown
- `--json` 提供稳定结构化输出，不改变信息层级
- 错误写入 stderr，结果写入 stdout
- 长任务进度使用现有 `...` 语气，不发明新的进度格式
- 成功继续使用 `✓` 前缀，警告继续使用 `⚠` 前缀
- retrieval outputs 与现有 `get/search/query` 风格一致，不引入“另一个 mini product”

Inheritance rule:
- `attach`, `detach`, `files`, `describe`, `import-chatgpt`, `imports` must feel native to the current CLI
- any new MCP response shape must mirror the same summary / next_action / details hierarchy

### Terminal Responsiveness & Accessibility

v0.5 is CLI-first. Responsive behavior means terminal-width adaptation, not CSS breakpoints.

Rules:
- default human output must remain readable at **80 columns**
- when a table would overflow, switch to stacked row rendering instead of truncating the most important field
- color may enhance meaning but may not be the only signal, status words must remain explicit
- `next_action` must always be copy-paste safe as a full command on one line
- warning and success glyphs (`⚠`, `✓`) must be paired with plain words for no-color / assistive contexts
- JSON / MCP output must never depend on terminal width and must expose the same semantics

Terminal behavior:

```text
wide terminal
  summary line
  compact table

narrow terminal (<= 80 cols)
  summary line
  next_action line
  stacked item blocks
```

Accessibility examples:
- prefer `Run completed with errors` over color-only red styling
- prefer `Next: gbrain describe --all` as a standalone line
- never hide retryability only behind icons or column alignment

Search result grouping rule:
- default human output groups by `result_kind`
- group order is `page` first, `file` second
- each group sorts by descending relevance
- JSON / MCP output preserves explicit `result_kind` and relevance scores, while also exposing grouped presentation metadata for clients that want the same UX
- if one group is empty, omit the heading instead of rendering an empty section

### What already exists

- `DESIGN.md` already defines CLI-first product posture, human-readable defaults, `--json`, stderr/stdout separation, and success/warning language
- existing `get/search/query` commands already establish the product's retrieval voice and should be extended, not reinvented
- existing command help style in `src/cli.ts` is short, direct, and usage-first

### Design NOT in scope

- no browser UI or dashboard in v0.5, this review only covers CLI + MCP user experience
- no visual mockups for this feature, because there is no page-level UI to render
- no motion, marketing hierarchy, or layout-system work in this version

### Non-Goals for v0.5

- 不做 PDF 内容提取
- 不做音频 / 视频 transcript
- 不做通用目录自动理解
- 不把文件系统设计绑死在 ChatGPT export 结构上
- 不在 v0.5 承诺第三方 importer SDK，只要求能自然接第二个 adapter

## Step 1: Schema Changes (src/core/db.ts)

### 新增到 SCHEMA 常量末尾

```sql
-- ── Files ────────────────────────────────────────────────────────────────────

CREATE TABLE IF NOT EXISTS files (
  slug          TEXT    PRIMARY KEY,
  sha256        TEXT    NOT NULL,
  file_path     TEXT    NOT NULL,
  original_name TEXT,
  mime_type     TEXT    NOT NULL,
  size_bytes    INTEGER NOT NULL,
  description   TEXT,
  created_at    TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_files_sha256 ON files(sha256);

CREATE TABLE IF NOT EXISTS page_files (
  page_slug     TEXT    NOT NULL REFERENCES pages(slug) ON DELETE CASCADE,
  file_slug     TEXT    NOT NULL REFERENCES files(slug) ON DELETE CASCADE,
  display_order INTEGER NOT NULL DEFAULT 0,
  created_at    TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  PRIMARY KEY (page_slug, file_slug)
);

CREATE INDEX IF NOT EXISTS idx_page_files_page ON page_files(page_slug);
CREATE INDEX IF NOT EXISTS idx_page_files_file ON page_files(file_slug);

-- Provenance / importer-specific references. Kept separate from page_files so one file
-- can be referenced multiple times within the same page/import source without losing detail.
CREATE TABLE IF NOT EXISTS file_references (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  page_slug     TEXT    NOT NULL REFERENCES pages(slug) ON DELETE CASCADE,
  file_slug     TEXT    NOT NULL REFERENCES files(slug) ON DELETE CASCADE,
  source_type   TEXT    NOT NULL,  -- "chatgpt", future importers
  source_ref    TEXT    NOT NULL,  -- conversation ID / import path / source-level ref
  source_item_id TEXT    NOT NULL DEFAULT '', -- message/node/file pointer ID
  source_role   TEXT,              -- "user" | "assistant" | "tool"
  created_at    TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  UNIQUE(page_slug, file_slug, source_type, source_ref, source_item_id)
);

CREATE INDEX IF NOT EXISTS idx_file_references_page ON file_references(page_slug);
CREATE INDEX IF NOT EXISTS idx_file_references_file ON file_references(file_slug);
CREATE INDEX IF NOT EXISTS idx_file_references_source ON file_references(source_type, source_ref);

-- Import runs: durable importer state, survives process exit and supports resume/reporting
CREATE TABLE IF NOT EXISTS import_runs (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  source_type     TEXT    NOT NULL,  -- "chatgpt", future importers
  source_ref      TEXT    NOT NULL,  -- input dir / file / source ref
  status          TEXT    NOT NULL,  -- "running" | "completed" | "completed_with_errors" | "failed" | "interrupted"
  total_items     INTEGER NOT NULL DEFAULT 0,
  completed_items INTEGER NOT NULL DEFAULT 0,
  failed_items    INTEGER NOT NULL DEFAULT 0,
  started_at      TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  finished_at     TEXT,
  summary         TEXT    NOT NULL DEFAULT ''
);

CREATE INDEX IF NOT EXISTS idx_import_runs_source ON import_runs(source_type, source_ref);
CREATE INDEX IF NOT EXISTS idx_import_runs_status ON import_runs(status, started_at);

CREATE TABLE IF NOT EXISTS import_run_items (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  import_run_id  INTEGER NOT NULL REFERENCES import_runs(id) ON DELETE CASCADE,
  item_key       TEXT    NOT NULL,  -- idempotency/resume unit, e.g. conversation ID
  item_type      TEXT    NOT NULL,  -- "conversation", future source units
  status         TEXT    NOT NULL,  -- "pending" | "completed" | "failed" | "skipped"
  page_slug      TEXT,
  error_code     TEXT,
  error_message  TEXT,
  retryable      INTEGER NOT NULL DEFAULT 0,
  updated_at     TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  UNIQUE(import_run_id, item_key)
);

CREATE INDEX IF NOT EXISTS idx_import_run_items_run ON import_run_items(import_run_id, status);

-- Source-level checkpoints: cross-run resume identity lives here.
-- import_run_items is execution history for one run; checkpoints are the durable source truth.
CREATE TABLE IF NOT EXISTS import_checkpoints (
  source_type    TEXT    NOT NULL,
  source_ref     TEXT    NOT NULL,
  item_key       TEXT    NOT NULL,
  item_type      TEXT    NOT NULL,
  status         TEXT    NOT NULL,  -- "completed" | "failed"
  page_slug      TEXT,
  last_run_id    INTEGER REFERENCES import_runs(id) ON DELETE SET NULL,
  updated_at     TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  PRIMARY KEY (source_type, source_ref, item_key)
);

CREATE INDEX IF NOT EXISTS idx_import_checkpoints_run ON import_checkpoints(last_run_id);
CREATE INDEX IF NOT EXISTS idx_import_checkpoints_source ON import_checkpoints(source_type, source_ref, status);

-- FTS5 index for file descriptions
CREATE VIRTUAL TABLE IF NOT EXISTS fts_files USING fts5(
  description,
  content='files',
  content_rowid='rowid',
  tokenize='porter unicode61'
);

-- Only index when description is not null
CREATE TRIGGER IF NOT EXISTS files_ai AFTER INSERT ON files
WHEN new.description IS NOT NULL
BEGIN
  INSERT INTO fts_files(rowid, description) VALUES (new.rowid, new.description);
END;

CREATE TRIGGER IF NOT EXISTS files_au AFTER UPDATE OF description ON files
WHEN new.description IS NOT NULL
BEGIN
  INSERT INTO fts_files(fts_files, rowid, description)
    VALUES('delete', old.rowid, COALESCE(old.description, ''));
  INSERT INTO fts_files(rowid, description) VALUES (new.rowid, new.description);
END;

CREATE TRIGGER IF NOT EXISTS files_ad AFTER DELETE ON files BEGIN
  INSERT INTO fts_files(fts_files, rowid, description)
    VALUES('delete', old.rowid, COALESCE(old.description, ''));
END;

-- Separate table for file description embeddings (content_chunks.page_id is NOT NULL,
-- cannot be reused for files without a table recreate migration — too risky)
CREATE TABLE IF NOT EXISTS file_chunks (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  file_slug   TEXT    NOT NULL REFERENCES files(slug) ON DELETE CASCADE,
  chunk_index INTEGER NOT NULL DEFAULT 0,
  chunk_text  TEXT    NOT NULL,
  embedding   BLOB,
  model       TEXT    NOT NULL DEFAULT 'text-embedding-3-small',
  embedded_at TEXT,
  created_at  TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  UNIQUE(file_slug, chunk_index)
);

CREATE INDEX IF NOT EXISTS idx_file_chunks_file ON file_chunks(file_slug);
```

export const MIME_TYPES: Record<string, string> = {
  '.jpg': 'image/jpeg', '.jpeg': 'image/jpeg', '.png': 'image/png',
  '.gif': 'image/gif', '.webp': 'image/webp', '.svg': 'image/svg+xml',
  '.pdf': 'application/pdf', '.mp4': 'video/mp4', '.m4a': 'audio/mp4',
  '.mp3': 'audio/mpeg', '.wav': 'audio/wav', '.heic': 'image/heic',
  '.tiff': 'image/tiff', '.tif': 'image/tiff',
  '.doc': 'application/msword',
  '.docx': 'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
};

export function getMimeType(filePath: string): string | null {
  return MIME_TYPES[extname(filePath).toLowerCase()] ?? null;
}

export function fileHash(content: Buffer): string {
  return createHash('sha256').update(content).digest('hex');
}

// description → kebab slug (max 80 chars), fallback to original_name
export function toFileSlug(text: string): string {
  return text
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, '-')
    .replace(/^-+|-+$/g, '')
    .slice(0, 80);
}

// Find non-colliding filename: react-diagram.png → react-diagram-2.png → ...
export function resolveFileSlug(
  desiredSlug: string,
  ext: string,
  filesDir: string
): string {
  let candidate = desiredSlug;
  let attempt = 1;
  while (existsSync(join(filesDir, `${candidate}${ext}`))) {
    attempt++;
    candidate = `${desiredSlug}-${attempt}`;
    if (attempt > 999) throw new Error(`Too many slug collisions for: ${desiredSlug}`);
  }
  return candidate;
}

export async function describeImage(
  base64Data: string,
  mimeType: string,
  cfg: VisionConfig,
  fetchFn: FetchFn = fetch
): Promise<string> {
  const baseUrl = cfg.base_url ?? "https://api.openai.com/v1";
  const model = cfg.model;
  const apiKey = cfg.api_key;

  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), 60_000); // 60s for images

  try {
    const res = await fetchFn(`${baseUrl}/chat/completions`, {
      method: "POST",
      headers: {
        "Authorization": `Bearer ${apiKey}`,
        "Content-Type": "application/json",
      },
      body: JSON.stringify({
        model,
        messages: [{
          role: "user",
          content: [
            {
              type: "text",
              text: "Describe this image in 2-3 sentences. Focus on the main subject, key visual elements, and any text visible. Be specific and informative — this description will be used as searchable text.",
            },
            {
              type: "image_url",
              image_url: { url: `data:${mimeType};base64,${base64Data}` },
            },
          ],
        }],
        max_tokens: 300,
      }),
      signal: controller.signal,
    });

    clearTimeout(timer);

    if (!res.ok) {
      const body = await res.text().catch(() => "");
      throw new Error(`Vision API error ${res.status}: ${body.slice(0, 200)}`);
    }

    const json = await res.json() as { choices?: Array<{ message?: { content?: string } }> };
    const content = json.choices?.[0]?.message?.content;
    if (!content) throw new Error("Vision API returned empty content");
    return content.trim();
  } catch (err) {
    clearTimeout(timer);
    if (err instanceof Error && err.name === "AbortError") {
      throw new Error("Vision API timeout (60s exceeded)");
    }
    throw err;

```typescript
searchKeyword(query: string, opts?: SearchOpts): SearchResult[] {
  const limit = opts?.limit ?? 10;
  const results: SearchResult[] = [];

  // Existing: page FTS
  try {
    const ftsRows = ftsSearch(this.db, query, { type: opts?.type, limit });
    for (const r of ftsRows) {
      results.push({
        slug: r.slug, page_id: r.id, title: r.title, type: r.type,
        chunk_text: r.snippet ?? '', chunk_source: 'compiled_truth' as const,
        score: r.score, stale: false, snippet: r.snippet,
      });
    }
  } catch { /* FTS error, continue */ }

  // New: file description FTS
  try {
    type FtsFileRow = {
      file_slug: string;
      title: string;
      description: string;
      rank: number;
      parent_page_slug: string | null;
      provenance_summary: string | null;
    };
    const fileRows = this.db.query<FtsFileRow, [string]>(
      `SELECT
         f.slug AS file_slug,
         COALESCE(f.original_name, f.slug) AS title,
         f.description,
         bm25(fts_files) AS rank,
         MIN(pf.page_slug) AS parent_page_slug,
         MIN(fr.source_type || ':' || fr.source_ref) AS provenance_summary
        FROM fts_files fts
        JOIN files f ON f.rowid = fts.rowid
        LEFT JOIN page_files pf ON pf.file_slug = f.slug
        LEFT JOIN file_references fr ON fr.file_slug = f.slug
        WHERE fts_files MATCH ?
        GROUP BY f.slug
        ORDER BY rank LIMIT ?`
    ).all(query, limit);

    for (const fr of fileRows) {
      results.push({
        result_kind: 'file',
        slug: `file:${fr.file_slug}`,
        page_id: -1,
        file_slug: fr.file_slug,
        parent_page_slug: fr.parent_page_slug ?? undefined,
        title: fr.title,
        type: 'file',
        chunk_text: fr.description ?? '',
        chunk_source: 'file_description' as const,
        provenance_summary: fr.provenance_summary ?? undefined,
        score: Math.abs(fr.rank),
        stale: false,
        snippet: fr.description?.slice(0, 200),
      });
    }
  } catch { /* FTS files error, continue */ }

### gbrain describe (src/commands/describe.ts)

```
args:
  file-slug: positional, optional (required unless --all)
  --all: boolean
  --force: boolean (re-describe already-described files)
  --max-files: number
  --concurrency: number (default 5)
  --max-bytes: number
  --db: string

flow for single:
  1. engine.getFile(slug) → if null, error (exit 1)
  2. loadConfig → check vision.api_key; if missing: error
     "✗ Vision API not configured. Set with: gbrain config set vision.api_key <key>" (exit 1)
  3. If mime_type is not image/* → error "✗ {slug} is not an image file ({mime_type}). describe only supports image/* in v0.5." (exit 1)
  4. If description exists AND NOT --force → print "· Already described. Use --force to override."
  5. Print "Describing {slug}..." (this is a blocking wait; 60s timeout possible)
  6. Call shared processImageFile() pipeline in "existing file" mode
  7. processImageFile() owns describeImage() + setFileDescription() + upsertFileChunk()
  8. Print: ✓ Described {slug}: "{description.slice(0, 60)}..."
     Next: gbrain search "{first 3 words of description}"

flow for --all:
  1. engine.listUndescribedFiles() (or all if --force)
  2. Filter by image/* only, `--max-files`, `--max-bytes`
  3. Print preflight summary: "X image candidates, Y skipped (size/type)"
  4. If running in TTY AND NOT --yes: prompt "Describe X images? [y/N] " and wait for confirm
     If NOT TTY OR --yes: proceed without prompt
  5. Batch with configurable concurrency (`--concurrency`, default 5)
  6. Retry retryable provider failures up to 2 times with backoff
  7. Progress in TTY mode: single updating line "Describing {X}/{Y}..." (ANSI cursor rewrite)
     Progress in non-TTY mode: new line every 10 completions "Describing {X}/{Y}..."
  8. Each file: same flow as single
  9. On SIGINT: print "Interrupted after X/Y. Resume with: gbrain describe --all"
  10. Final: "✓ Described X files, Y failed, Z skipped"
      If Y > 0: "Next: gbrain describe --all (retries failed files)"

extra flags:
  --yes: skip TTY confirm prompt (for scripting)
```

### gbrain import-chatgpt (src/commands/import-chatgpt.ts)

```
args:
  export-dir: positional, required
  --dry-run: boolean
  --describe: boolean (trigger gbrain describe on imported images)
  --resume: number | "latest"
  --concurrency: number
  --max-images: number
  --db: string

flow:
  1. Verify export-dir exists
  2. Supported input for v0.5:
     - unpacked ChatGPT export directory containing `conversations-*.json` + attachments
     - NOT supported: zip file directly, single JSON file without attachment directory
     - If zip detected (export-dir ends in .zip or is a zip file): error
       "✗ {path} appears to be a zip file. Unpack it first: unzip {path} -d {dir}"  (exit 1)
     - If export-dir exists but no conversations-*.json found: error
       "✗ No conversations found. Expected a ChatGPT export directory with conversations-*.json" (exit 1)
  3. Build file index: scan export-dir → Map<fileId, string> (filePath)
     Pattern A: "file-{fileId}-{name}.{ext}" → extract fileId
     Pattern B: "{hash}#file_{numericId}#{page}.{hash}.{ext}" → index by numericId
  4. Find conversations-*.json files (glob export-dir/conversations-*.json)
  5. This command is a **source adapter**, not the file system itself.
     It must reuse the generic attach/describe pipeline rather than duplicating file logic.
  6. Create or resume `import_run`
  7. Adapter parses source into generic `AsyncIterable<ImportUnit>`
  8. Runner processes each ImportUnit using `item_key = conversation_id`
     and checks `import_checkpoints(source_type, source_ref, item_key)` before work
  9. For each JSON file:
      a. JSON.parse — wrap in try/catch per file, skip + warn on parse error
      b. For each conversation:
          - slug = "chatgpt-{conversation_id.slice(0,8)}"
         - Extract title, messages (tree traversal via mapping)
         - Build content: chronological messages as markdown
         - Extract image refs: parts where content_type = "image_asset_pointer"
           → asset_pointer = "file-service://file-{fileId}" → extract fileId
         - engine.putPage(slug, { type: 'session', title, compiled_truth: content })
         - For each image ref: fileIndex.get(fileId) → if found, call engine.attachFile(...)
         - Then call engine.recordFileReference(...) with:
           source_type="chatgpt"
           source_ref=conversation_id
           source_item_id=node_id or asset_pointer
           source_role=message.author.role
          - If --describe: call shared processImageFile() for image/* attachments only
          - Mark import_run_item completed / failed / retryable
          - Upsert source-level checkpoint after per-unit commit
  10. Print canonical 3-layer completion output (see Canonical Output Examples above)
      (even in dry-run, just print what WOULD happen)

resume behavior:
  --resume {id}: load existing run, skip items where import_checkpoints record exists
  --resume latest: find most recent run for this source_ref
    If latest run status = "completed": print "⚠ Run {id} is already completed. Starting new run." and create fresh run
    If latest run status = "in_progress" or "completed_with_errors": resume it

scope:
  - v0.5 的 import-chatgpt 只负责 ChatGPT export 的扫描和图片挂接
  - 通用文件系统不得依赖 ChatGPT 的目录结构或字段命名
  - importer framework 负责 run tracking / idempotency / reporting
  - 后续其他 importer 应复用相同的 files / page_files / describe / search integration

compatibility:
  - existing `gbrain digest` remains as a compatibility shim in v0.5
  - shim behavior: warn that `digest` is deprecated, support the overlapping ChatGPT import subset only,
    and error clearly when old-only flags/inputs cannot be mapped to `import-chatgpt`

ChatGPT message tree traversal:
  The conversations have a `mapping` field (Record<nodeId, {message, parent, children}>).
  To get chronological messages:
  1. Find root node (parent = null OR parent = "client-created-root")
  2. Walk children depth-first, collect message.content items
  3. message.author.role = "user" | "assistant" | "tool"
  4. Include both user and assistant messages

File index building:
  const fileIndex = new Map<string, string>();
  for (const name of readdirSync(exportDir)) {
    // Pattern A: file-{fileId}-anything.ext
    const matchA = name.match(/^file-([A-Za-z0-9]+)-/);
    if (matchA) fileIndex.set(matchA[1], join(exportDir, name));
    // Pattern B: hash#file_{numericId}#page.hash.ext
    const matchB = name.match(/#file_(\d+)#/);
    if (matchB) fileIndex.set(matchB[1], join(exportDir, name));
  }

Critical: handle missing fields gracefully. Some conversations may lack title,
created_at, etc. Always use optional chaining + defaults.
```

## Critical Gaps (MUST fix during implementation)

### Gap 1: import-chatgpt JSON parse errors

**Problem:** If any `conversations-*.json` has malformed JSON or missing fields, the whole import must not crash.

**Fix:** Wrap per-conversation processing in try/catch:
```typescript
let importErrors = 0;
for (const conv of conversations) {
  try {
    // ... process conversation
  } catch (err) {
    importErrors++;
    console.warn(`⚠ Skipped conversation ${conv.id ?? '?'}: ${err.message}`);
  }
}
```

### Gap 2: FTS5 trigger safety in setFileDescription

**Problem:** If `setFileDescription()` is called on a file that doesn't exist (stale reference), the UPDATE triggers will operate on 0 rows silently.

**Fix (already in spec above):** Check `rowid` existence before UPDATE, throw if not found. This is already in the `setFileDescription()` spec.

### Gap 3: import_run bootstrap failure

**Problem:** If `createImportRun()` throws (DB write failure), the importer has no run ID and cannot track progress. The current spec doesn't specify what happens in this edge case.

**Fix:** Wrap `createImportRun()` in try/catch at the top of the import command. If it throws, print a clear error and exit:
```typescript
let runId: number;
try {
  runId = engine.createImportRun({ source_type: "chatgpt", source_ref: absDir, total_items: conversations.length });
} catch (err) {
  console.error(`✗ Could not create import run: ${err.message}`);
  process.exit(1);
}
```

### Gap 4: orphaned file detection

Add `--orphaned` flag to the `files` command to surface files whose source path no longer exists on disk:

```typescript
// In listFiles / files command, when --orphaned is passed:
// For each file in DB, check if join(filesDir, file.file_path) exists on disk
// Report missing files with their page associations
```

This prevents silent data loss after source files are moved or deleted.
