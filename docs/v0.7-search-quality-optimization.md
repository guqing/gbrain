## 优先级 1 — 召回率修复（最高影响）

### 1.1 中文 FTS 分词：从 unicode61 迁移到 trigram

**问题**：unicode61 tokenizer 将中文按单字拆分，导致：
- "git 提交邮箱修改" → 提、交、邮、箱、修、改（6 个单字 token）
- "修改Git提交邮箱" 页面的单字 token 在其他无关页面中出现频率更高
- BM25 评分无法区分"恰好包含这些字"和"真正讨论这个话题"

**解决方案：trigram tokenizer（SQLite 3.44+ 内置）**

```sql
-- 重建 FTS 表，使用 trigram tokenizer
CREATE VIRTUAL TABLE page_fts_v2 USING fts5(
  title,
  compiled_truth,
  content='pages',
  content_rowid='id',
  tokenize="trigram"
);
```

**trigram 原理**：将文本切成连续的 3 字符 n-gram，例如：
- "提交邮箱" → "提交邮" "交邮箱"
- 查询"提交邮箱"时，搜索包含"提交邮"和"交邮箱"的文档

**对比实验（中文 SQLite FTS5）**：

| Tokenizer | 查询"git 提交邮箱修改" | 索引大小 | 查询速度 |
|-----------|----------------------|---------|---------|
| unicode61 | ❌ 找不到 | 1x | 快 |
| trigram | ✅ 精确召回 | 3-4x | 较快 |
| ICU | ✅ 词边界分词 | 1.5x | 需要 ICU 扩展 |

trigram 是最佳实践选择，因为：
1. SQLite 3.44+ 内置，无需额外依赖
2. 对所有语言有效（英文、中文、日文、混合）
3. 不需要语言检测或词典

**迁移方案（不破坏现有数据）**：

```typescript
// src/core/db.ts — migrateDb() 末尾追加

// 检查当前 FTS 是否已使用 trigram
const ftsInfo = db.query<{tokenize: string}, []>(
  "SELECT tokenize FROM fts5_tokenize WHERE table_name='page_fts'"
).get();

// 注意：fts5_tokenize 是内部表，实际检查方式：
const hasTrigramFts = db.query<{name: string}, []>(
  "SELECT name FROM sqlite_master WHERE type='table' AND name='page_fts_v2'"
).get();

if (!hasTrigramFts) {
  db.exec(`
    CREATE VIRTUAL TABLE IF NOT EXISTS page_fts_v2 USING fts5(
      title,
      compiled_truth,
      content='pages',
      content_rowid='id',
      tokenize="trigram"
    );
    INSERT INTO page_fts_v2(page_fts_v2) VALUES('rebuild');
  `);
}
```

**更新 fts.ts 使用新表**：

```typescript
// src/core/fts.ts — searchKeyword() 函数

// 改为使用 page_fts_v2（trigram），同时保留 page_fts（unicode61）作为 fallback
const rows = db.query<FtsRow, [string]>(`
  SELECT
    p.slug, p.title, p.type,
    snippet(page_fts_v2, 1, '', '', '...', 24) as snippet
  FROM page_fts_v2
  JOIN pages p ON page_fts_v2.rowid = p.id
  WHERE page_fts_v2 MATCH ?
  ORDER BY rank
  LIMIT 20
`).all(query);
```

**trigram 注意事项**：
- 索引大小膨胀 3-4x（对个人知识库规模不是问题）
- 不支持 prefix queries（`term*`），但对我们的使用场景影响小
- `snippet()` 在 trigram 表上正常工作

### 1.2 跨语言召回：LIKE fallback + 向量兜底

**问题**："PostgreSQL 自然排序" 找不到 "PostgreSQL Natural Sorting"，因为：
1. 该页面零 embedding（没有运行 `exo embed`）
2. FTS BM25 无法桥接中英文

**解决方案组合**：

**Step A：确保所有页面都有 embedding（`exo embed` 应该是 ingest 的一部分）**

```typescript
// src/commands/ingest.ts（或 add.ts）— 在 compile 之后自动触发 embed
// 保证每个新导入的页面都有 embedding，不需要用户手动跑 exo embed
```

**Step B：Title LIKE 搜索作为额外信号**

```typescript
// src/core/fts.ts — 在 FTS 搜索后追加 title LIKE 搜索

function titleLikeSearch(db: Database, query: string): SearchResult[] {
  const terms = query.split(/\s+/).filter(t => t.length >= 2);
  if (terms.length === 0) return [];
  
  // Build LIKE conditions for each term (case-insensitive)
  const conditions = terms.map(() => "lower(p.title) LIKE lower(?)").join(" AND ");
  const params = terms.map(t => `%${t}%`);
  
  return db.query<SearchResult, string[]>(`
    SELECT p.slug, p.title, p.type, '' as snippet
    FROM pages p
    WHERE ${conditions}
    LIMIT 10
  `).all(...params);
}
```

这个方案能捕获 "PostgreSQL Natural Sorting" 当查询包含 "postgresql" 时（大小写不敏感）。

## 优先级 2 — 向量幽灵结果

**问题**：向量搜索返回语义相近但内容无关的结果（如查 "redis 限流" 返回 "gemini 3 migration"），且这些结果的 snippet 为空。

**解决方案：向量最低相似度阈值**

在 `sqlite-engine.ts` 的 `searchVector()` 中，过滤掉相似度低于阈值的结果：

```typescript
// src/core/sqlite-engine.ts — searchVector()

const MIN_COSINE_SIMILARITY = 0.3; // 余弦相似度 0-1，低于此值不认为相关

// 在 sqlite-vec 查询中加 HAVING 或后处理过滤
const results = vectorQueryResults.filter(r => r.similarity >= MIN_COSINE_SIMILARITY);
```

这个阈值需要实验确定。0.3 是起始值，可以通过评估框架调整。

## 优先级 3 — 重复标题区分

**问题**：两个同名会话（如"抓包工具推荐"）无法区分

**解决方案**：在 title 后追加来源日期或 ID 后缀（仅当有重复时）

```typescript
// src/commands/query.ts — 结果渲染阶段

// 检测重复 title
const titleCounts = new Map<string, number>();
for (const r of results) titleCounts.set(r.title, (titleCounts.get(r.title) ?? 0) + 1);

// 渲染时对重复 title 加日期后缀
const display = titleCounts.get(r.title)! > 1
  ? `${r.title} (${r.slug.slice(-8)})`  // 或显示 created_at 日期
  : r.title;
```
