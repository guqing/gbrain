# exo v0.6 — Unified Content Extraction

## Problem

v0.5 could only extract meaning from images via the Vision API.  PDFs, Word
documents, audio recordings, and video files were attached and stored but
remained opaque — unsearchable, unembedded, invisible to `query`.

## Goal

A single processing pipeline that extracts text/descriptions from every supported
file type, stores chunks in the DB, embeds them, and surfaces them through the
existing hybrid search interface.

---

## Architecture

```
                          ┌──────────────────────────────┐
File on disk              │        processFileContent()    │
(PDF / DOCX / img /  ───▶ │                              │
 audio / video)           │  1. ExtractorRegistry.get()  │
                          │  2. extractor.extract()       │
                          │  3. embedTexts() [batch]     │
                          │  4. upsertFileChunks()        │
                          │  5. setProcessedAt()          │
                          └──────────────────────────────┘
                                       │
              ┌────────────────────────┼──────────────────────────┐
              │                        │                          │
      file_chunks rows          files.processed_at        fts_file_chunks
      (text + embeddings)         (ISO timestamp)          (FTS5 index)
              │                                                    │
              └──────────┬─────────────────────────────────────────┘
                         │
                    searchVector()  +  searchKeyword()
                         │
                       query results with chunk_source labels
```

### Extractor registry

`ExtractorRegistry` maps MIME types to `ContentExtractor` implementations.
Five extractors are auto-registered at startup:

| Extractor | MIME types | Output |
|-----------|-----------|--------|
| `ImageExtractor` | `image/*` | 1 chunk, `chunk_source = "description"` |
| `PdfExtractor` | `application/pdf` | 1 chunk per page, `chunk_source = "page-N"` |
| `DocumentExtractor` | DOCX | 1 chunk per paragraph, `chunk_source = "paragraph-N"` |
| `AudioExtractor` | `audio/*` | 1 chunk (full transcript), `chunk_source = "transcript"` |
| `VideoExtractor` | `video/*` | extracts audio via ffmpeg → delegates to AudioExtractor |

## Database schema additions (v0.6)

### `file_chunks.chunk_source`

```sql
ALTER TABLE file_chunks ADD COLUMN chunk_source TEXT NOT NULL DEFAULT 'description';
```

Identifies the origin of each chunk within a file:
- `"description"` — AI image description (v0.5 compat)
- `"page-1"`, `"page-2"`, … — PDF page
- `"paragraph-1"`, … — DOCX paragraph
- `"transcript"` — Whisper audio/video transcription

### `files.processed_at`

```sql
ALTER TABLE files ADD COLUMN processed_at TEXT;
```

NULL = not yet processed. ISO-8601 timestamp = processed.  Replaces the v0.5
`description IS NULL` heuristic as the "needs processing" sentinel.

Back-fill on migration:
```sql
UPDATE files SET processed_at = created_at WHERE description IS NOT NULL;
```

### `fts_file_chunks` FTS5 index

```sql
CREATE VIRTUAL TABLE fts_file_chunks USING fts5(
  chunk_text,
  content='file_chunks',
  content_rowid='id',
  tokenize='porter unicode61'
);
```

Three sync triggers (`file_chunks_ai`, `file_chunks_au`, `file_chunks_ad`) keep it
in sync with `file_chunks`.  `searchKeyword()` UNION-joins results from both
`fts_files` and `fts_file_chunks`.

## Key data types

```typescript
// ContentChunk — one searchable unit from a file
interface ContentChunk {
  text: string;
  source: string;   // "page-1", "transcript", "description", …
}

// ExtractOpts — passed to every extractor
interface ExtractOpts {
  visionConfig?: VisionConfig;
  transcriptionBaseUrl?: string;
  transcriptionApiKey?: string;
  fetchFn?: typeof fetch;   // injectable for tests
}

// ContentExtractor — all extractors implement this
interface ContentExtractor {
  supportedMimeTypes: string[];
  supports(mimeType: string): boolean;
  extract(filePath: string, mimeType: string, opts: ExtractOpts): Promise<ContentChunk[]>;
}
```

## Extractor details

### PdfExtractor (`unpdf`)

Uses `unpdf@1.4.0` (Bun-compatible, no native bindings).  Calls
`extractText(doc)` once; the returned `text[]` array maps index → page number.
One `ContentChunk` per non-empty page.

### DocumentExtractor (`mammoth`)

Uses `mammoth@1.12.0`.  Calls `extractRawText({ buffer })`, splits on double
newlines to produce paragraph chunks.

### AudioExtractor (Whisper API)

Sends the raw audio file as `multipart/form-data` to the configured
`transcription.base_url` (defaults to vision or embed config, then
`OPENAI_API_KEY`).  Returns a single `"transcript"` chunk.

Limits:
- File size ≥ 25 MB → throws with a clear message
- Request timeout: 2 minutes via `AbortSignal.timeout(120_000)`

### VideoExtractor (ffmpeg → AudioExtractor)

Spawns `ffmpeg -i <input> -f wav -ar 16000 -ac 1 -` and pipes stdout to a temp
WAV file, then delegates to `AudioExtractor`.  Temp file is always cleaned up in
`finally`.  If `ffmpeg` is not installed, throws a user-friendly error:
`"ffmpeg not found — install ffmpeg to transcribe video"`.

### ImageExtractor (Vision API)

Thin wrapper over the existing `describeImage()` from `vision.ts`.
Returns one `"description"` chunk.  Also writes `files.description` for
backward-compat FTS (`fts_files`).

## Backward compatibility

- `describeAndEmbedFile()` is now a `@deprecated` thin adapter over
  `processFileContent()`.  Callers in the importer runner continue to work
  unchanged.
- `listUndescribedFiles()` delegates to `listUnprocessedFiles()`.
- `chunk_source` column defaults to `"description"` — existing rows are unaffected.
- `fts_files` (image description FTS) still populated via `setFileDescription()`.

## Configuration

```toml
[transcription]
base_url = "https://api.openai.com/v1"  # optional, falls back to vision or embed
api_key  = "sk-..."                      # optional, same fallback chain
```

Fallback chain for transcription credentials:
1. `transcription.api_key`
2. `vision.api_key`
3. `embed.api_key`
4. `OPENAI_API_KEY` environment variable

## CLI changes

### `exo describe`

- `--all` now uses `listUnprocessedFiles()` + `registry.supports()` filter.
  PDF, DOCX, images, and audio/video files are all included.
- Single-file mode checks `files.processed_at` instead of `files.description`.

### `exo attach`

New flags:

| Flag | Meaning |
|------|---------|
| `--describe` | Generate AI description for images (existing, unchanged) |
| `--transcribe` | Transcribe audio/video using Whisper API |
| `--no-embed` | Store the file only, skip content extraction |

PDF and DOCX are auto-extracted on attach (no flag required).

### `exo query`

Results now show the chunk source label for non-image file results:

```
[0.016] my-document  (file)  [page-3]
  The third section discusses risk management frameworks and their...
```

## Design decisions

**Why `processed_at` instead of `description IS NULL`?**
PDF/DOCX/audio files never get a `description` field.  A single nullable
timestamp works for all file types and makes "needs reprocessing" logic
straightforward.

**Why per-page chunks for PDF instead of one big blob?**
Granular chunks produce more relevant vector search hits.  A 100-page PDF
with one embedding is noise; 100 embeddings allow pinpointing the relevant page.

**Why delegate video → audio via AudioExtractor instead of Whisper's video support?**
The Whisper API only accepts audio formats.  ffmpeg normalises the audio track to
WAV (16 kHz mono) before the API call.  This keeps `AudioExtractor` reusable.

**Why injectable `fetchFn` in AudioExtractor?**
Allows unit-testing transcription logic without real HTTP calls.  The default is
the global `fetch`.

**Why `upsertFileChunks` uses delete-then-insert?**
Re-processing a file (after upgrading the model or fixing a config) should replace
chunks cleanly.  A pure upsert-by-index risks leaving stale chunks if chunk count
decreases.
